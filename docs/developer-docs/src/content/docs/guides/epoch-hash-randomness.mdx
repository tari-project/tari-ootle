### Overview

Smart contract templates frequently need randomness that is:

1. **Deterministic** — every validator node must agree on the same value for a given transaction.
2. **Unpredictable in advance** — the value must not be knowable at the time a user commits to a
   bet or game outcome.
3. **Manipulation-resistant** — no single validator or small coalition should be able to bias the
   value.

Tari Ootle provides `Consensus::current_epoch_hash()` as a built-in randomness primitive that
satisfies all three properties. This guide explains how it works and how to use it in
epoch-based betting templates.

---

### How epoch hashes work

Every block in Tari Ootle carries an `epoch_hash` field in its header. This hash is derived from
the **L1 base-layer block hash** that triggered the epoch transition.

Key properties:

- **Fixed for the entire epoch.** Every block in epoch N carries the same `epoch_hash`. Two
  transactions executed in different blocks of the same epoch see the same value.
- **Unknown until the epoch begins.** The hash depends on which L1 block is mined at the epoch
  boundary — which is unpredictable before it happens.
- **Agreed upon by all validators.** It is part of the signed block header; any disagreement
  causes a consensus failure.

### Template API

```rust
use tari_template_lib::prelude::*;

pub fn my_function() {
    // Returns the epoch hash of the current epoch as a 32-byte array.
    let seed: [u8; 32] = Consensus::current_epoch_hash();

    // Use individual bytes as a lightweight random oracle.
    // Odd first byte → outcome 1; even → outcome 0.
    let outcome = seed[0] % 2;
}
```

`Consensus::current_epoch_hash()` is available alongside the existing `Consensus::current_epoch()`.

<Aside type="caution">
  For high-value applications, the epoch hash provides sufficient resistance against casual manipulation but is not as
  strong as a VRF-based randomness beacon. A dishonest leader can choose not to propose a block (at the cost of losing
  their leader reward), but cannot choose the epoch hash itself. Use this primitive for moderate-stakes use cases; for
  very high-value applications, wait for the dedicated VRF beacon (future work).
</Aside>

### The commit-reveal pattern

The canonical pattern for using epoch hashes in betting is a two-epoch commit-reveal:

1. **Epoch N — commit**: the player deposits their stake and the current epoch is recorded.
2. **Epoch N+1 or later — reveal**: anyone calls `settle`. The epoch hash of the settlement
   epoch — which was unknowable at commit time — is used as the randomness seed.

This guarantees that neither the player nor any validator knew the outcome when the bet was
placed.

```rust
pub fn place_bet(stake: Bucket, player_account: ComponentAddress) {
    // Record the epoch so we can enforce at least one epoch gap.
    let placement_epoch = Consensus::current_epoch();
    // ... store stake and placement_epoch in component state ...
}

pub fn settle(&mut self) {
    let current_epoch = Consensus::current_epoch();
    assert!(
        current_epoch > self.placement_epoch,
        "Cannot settle a bet in the same epoch it was placed"
    );

    // The seed was unknowable at placement time.
    let seed = Consensus::current_epoch_hash();
    let won = seed[0] % 2 == 1;
    // ... distribute funds ...
}
```

### The keeper pattern

Template execution on Tari Ootle is always user-initiated — there is no native scheduler that
can trigger settlement automatically once the next epoch begins. This creates a UX problem for
betting: the player must return to settle their own bet.

The **keeper pattern** solves this:

- The `settle` method is callable by **anyone**, not just the original bettor.
- A small **keeper fee** (e.g. 0.5%) is taken from the pot and paid to whoever calls `settle`.
- External **keeper bots** watch the chain for unsettled bets and race to settle them, earning
  the fee as compensation.

```rust
pub fn settle(&mut self, caller_account: ComponentAddress) {
    let is_keeper = caller_account != self.player_account;

    // Pay keeper fee before determining outcome.
    if is_keeper {
        let fee = compute_keeper_fee(self.stake_vault.balance());
        let fee_bucket = self.stake_vault.withdraw(fee);
        ComponentManager::get(caller_account).invoke("deposit", args![fee_bucket]);
    }

    // ... determine outcome and distribute remaining funds ...
}

fn compute_keeper_fee(total: Amount) -> Amount {
    // 0.5% fee, minimum 1 microtari.
    let fee = total * 50u64 / 10_000u64;
    fee.max(Amount::ONE)
}
```

### Reference implementation: house-based coin-flip

The reference implementation in `crates/engine/tests/templates/` uses a two-component
architecture that avoids burning tokens and allows the house to accumulate profit from losses.

#### `EpochBettingHouse`

A long-lived component holding a liquidity reserve. The operator deploys it with initial
capital and passes the address of the companion `EpochBet` template.

```rust
pub fn new(initial_reserve: Bucket, bet_template: TemplateAddress) -> Component<Self>
pub fn place_bet(&mut self, stake: Bucket, player_account: ComponentAddress) -> ComponentAddress
pub fn pay_winner(&mut self, player_account: ComponentAddress, prize_amount: Amount)
pub fn collect_loss(&mut self, stake: Bucket)
```

When `place_bet` is called, the house validates it has sufficient reserve to cover a potential
win (`reserve ≥ stake`) and creates a new `EpochBet` component via `TemplateManager`.

#### `EpochBet`

A short-lived, per-bet component holding the player's stake in escrow. Settlement cross-calls
the house:

- **Win** (seed byte odd): returns the player's stake from the vault and calls
  `house.pay_winner(player, stake)` to deliver the matching prize. The player receives
  `2 × stake` total.
- **Loss** (seed byte even): forwards the stake to the house reserve via
  `house.collect_loss(stake)`. The house accumulates profit; no tokens are burnt.

```rust
pub fn settle(&mut self, caller_account: ComponentAddress) {
    // ... keeper fee logic ...

    let seed = Consensus::current_epoch_hash();
    let won = seed[0] % 2 == 1;
    let stake_amount = self.stake_vault.balance();

    if won {
        let stake = self.stake_vault.withdraw_all();
        ComponentManager::get(self.player_account).invoke("deposit", args![stake]);
        ComponentManager::get(self.house)
            .invoke("pay_winner", args![self.player_account, stake_amount]);
    } else {
        let lost_stake = self.stake_vault.withdraw_all();
        ComponentManager::get(self.house).invoke("collect_loss", args![lost_stake]);
    }
}
```

<Aside type="tip">
  Buckets can be passed as arguments to cross-component `invoke` calls. Returning a `Bucket` from a cross-component call
  is not required and is avoided in this design — the house deposits directly into the player's account rather than
  returning a `Bucket` to the `EpochBet` caller.
</Aside>

### Testing epoch hash behaviour

The `TemplateTest` harness lets you inject arbitrary epoch hashes to test deterministic
outcomes:

```rust
use tari_engine_types::virtual_substate::{VirtualSubstate, VirtualSubstateId};
use tari_template_test_tooling::TemplateTest;

let mut test = TemplateTest::new(CRATE_PATH, TEMPLATE_PATHS);

// Force a specific epoch and hash.
test.set_virtual_substate(VirtualSubstateId::CurrentEpoch, VirtualSubstate::CurrentEpoch(2));
let mut win_hash = [0u8; 32];
win_hash[0] = 0x01; // odd → win
test.set_virtual_substate(
    VirtualSubstateId::CurrentEpochHash,
    VirtualSubstate::CurrentEpochHash(win_hash),
);

// To test the "missing virtual substate" error path:
test.remove_virtual_substate(VirtualSubstateId::CurrentEpochHash);
// The next transaction that calls current_epoch_hash() will now return
// Execution failure: Virtual substate not found: Virtual(CurrentEpochHash)
```

### Security considerations

| Threat                               | Mitigation                                                                                                                                             |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Player chooses settlement epoch      | Settlement is valid only from `placement_epoch + 1` onwards. The relevant epoch hash is always future at placement time.                               |
| Validator manipulates epoch hash     | The epoch hash is derived from the L1 block hash at the epoch boundary, which is fixed by L1 mining and not controllable by any single validator.      |
| Dishonest leader withholds block     | The leader forfeits their block reward; they cannot choose the epoch hash. Epoch-level granularity reduces the attack surface vs per-block randomness. |
| Keeper manipulates settlement timing | Keepers can only call `settle` after the settlement epoch begins. They cannot choose the epoch hash.                                                   |
