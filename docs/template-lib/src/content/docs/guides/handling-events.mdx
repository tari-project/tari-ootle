---
title: Handling Events
description: Emit and subscribe to custom events within your Tari Ootle Templates.
---

Events provide a powerful mechanism for your Templates to communicate with the outside world and for different parts of your decentralized application to react to state changes without direct method calls. The Tari Ootle Template Library supports custom event emission and handling.

### Emitting Events

Templates can emit events using the `emit_event!` macro. Events are typically structured data that describes something that has occurred within your component.

```rust
use tari_template_lib::prelude::*;
use tari_template_lib_types::ComponentAddress;

// Define a struct for your event data
#[derive(Debug, serde::Serialize, serde::Deserialize)]
struct MyCustomEvent {
    component_address: ComponentAddress,
    value: u64,
    message: String,
}

#[template]
mod MyEventEmitter {
    pub struct EventEmitter {
        counter: u64,
    }

    impl EventEmitter {
        pub fn new() -> Self {
            Self { counter: 0 }
        }

        pub fn increment_and_emit(&mut self, increment_by: u64) {
            self.counter += increment_by;

            let event = MyCustomEvent {
                component_address: tari_template_lib::component::current_component_address(),
                value: self.counter,
                message: format!("Counter incremented by {}", increment_by),
            };

            // Emit the event
            emit_event!(event);

            info!("Emitted MyCustomEvent with value: {}", self.counter);
        }
    }
}
```

#### Explanation:

-   **Event Data Structure:** Define a Rust `struct` that represents the data you want to include in your event. This struct **must** derive `serde::Serialize` and `serde::Deserialize` so it can be serialized into a format that can be stored on-chain and consumed by external systems.
-   `emit_event!(event)`: This macro takes an instance of your event struct and serializes it, then sends it to the Tari Ootle engine as an event.
-   `tari_template_lib::component::current_component_address()`: A utility function to get the address of the component currently executing. This is often useful to include in event data to identify the source.

### Event Metadata

When an event is emitted, the Tari Ootle engine automatically attaches additional metadata, such as:

-   **`component_address`**: The address of the component that emitted the event.
-   **`template_address`**: The address of the template from which the component was instantiated.
-   **`transaction_hash`**: The hash of the transaction that caused the event to be emitted.
-   **`block_hash`**: The hash of the block in which the transaction was included.
-   **`timestamp`**: The time the event occurred.
-   **`event_type`**: The Rust type name of the event struct (e.g., `"MyCustomEvent"`).

This metadata, combined with your custom event data, provides a rich source of information for off-chain observers.

### Subscribing to Events (Off-Chain)

While events are emitted by Templates on-chain, they are primarily consumed by off-chain applications, such as:

-   **Indexers:** Services that listen to blockchain events and store them in a queryable database.
-   **Wallets:** To update user interfaces in real-time when certain actions occur (e.g., token transfers).
-   **Monitoring Tools:** To alert developers to critical events or state changes.
-   **Other Decentralized Applications:** To react to events from one Template within another application.

The exact mechanism for subscribing to events will depend on the client-side SDKs or APIs provided by the Tari Ootle network (e.g., WebSocket connections to a full node or indexer service). Your off-chain application would typically filter events by `component_address`, `template_address`, or `event_type` and then deserialize the `data` payload back into your Rust event struct (or its equivalent in another language).

### Best Practices for Events

-   **Meaningful Data:** Include all necessary information in your event struct that an off-chain consumer would need to understand and react to the event.
-   **Immutability:** Events represent past occurrences; they should not be mutable after emission.
-   **Clear Naming:** Use descriptive names for your event structs and their fields.
-   **Logging vs. Events:** Use `info!`, `warn!`, `error!` macros for debugging and operational logging within the validator node. Use `emit_event!` for structured data that off-chain applications need to consume.

By strategically using events, you can build responsive and observable decentralized applications on Tari Ootle.
